### Request flow for a chat call
```
Frontend
  → ClaudeProxyController   (receives issue + messages)
  → RagSearchService        (top-8 chunks for the issue text)
  → Claude API              (system prompt = RAG chunks, user messages appended)
  → parse JSON response
  → Frontend
```

### Key design rules
- **JWT is required** on all endpoints — `JwtAuthFilter` rejects unauthenticated requests
- **SQL safety gate** — `QuerySafetyValidator` must pass before `DatabaseExecutorController` runs any query; only `SELECT` is allowed
- **RAG chunks** are stored in the DB via `RagChunk` entity; ingestion goes through `RagIngestionController`
- Config lives in `application.yml` (not `application.properties`)

## RAG Ingestion — File Upload & Chunking

### Accepted file types
| Extension | Classified as | Description |
|-----------|--------------|-------------|
| `.java`   | `java`       | Service / DAO / controller files |
| `.java` (name contains `Entity` or `Model`) | `entity` | Hibernate entity classes |
| `.xml`, `.hql` | `hql`   | HQL / XML query mapper files |
| `.ddl`    | `ddl`        | Database schema / DDL scripts |
| `.sql` (contains stored procedure markers) | `proc` | Stored procedures |
| anything else | `other`  | Fallback |

### Chunking strategy
- **Chunk size**: 1000 characters
- **Overlap**: 150 characters between consecutive chunks
- Each chunk stored as one row in `rag_chunk`

### `rag_chunk` table schema
```sql
CREATE TABLE rag_chunk (
  id           BIGINT AUTO_INCREMENT PRIMARY KEY,
  file_name    VARCHAR(255)  NOT NULL,
  file_type    VARCHAR(50)   NOT NULL,   -- java | entity | hql | ddl | proc | other
  chunk_index  INT           NOT NULL,   -- 0-based position within the file
  content      TEXT          NOT NULL,
  uploaded_at  DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FULLTEXT INDEX idx_content (content)   -- enables fast MATCH AGAINST search
);
```

> `FULLTEXT` index on `content` is required — `RagSearchService` uses `MATCH(content) AGAINST(?)` for chunk retrieval.

### Re-upload behaviour (idempotent)
- On upload, delete **all existing chunks** for that `file_name` first
- Then ingest fresh chunks — no duplicates, no stale data

### `RagIngestionController` endpoints
| Method | Path | Description |
|--------|------|-------------|
| `POST`   | `/api/rag/upload`         | Upload one or more files; auto-classify + chunk + store |
| `GET`    | `/api/rag/files`          | List all ingested file names and their chunk counts |
| `GET`    | `/api/rag/search?q=...`   | Full-text search across all chunks; returns top matches |
| `DELETE` | `/api/rag/files/{fileName}` | Delete all chunks for a given file |

- Backend parses the JSON response from Claude

### Key rules when modifying this flow

- Claude must **never** return markdown — only the raw JSON object above
- All 8 fields are required; do not add or remove fields without updating the frontend parser
- `sql` must be a **complete, executable** MySQL `SELECT` query (no placeholders)
- `severity` must be one of the four exact enum values: `LOW | MEDIUM | HIGH | CRITICAL`

## Database Setup Scripts

### Read-only application user
```sql
CREATE USER 'dms_readonly'@'%' IDENTIFIED BY 'StrongPassword123!';
GRANT SELECT ON dms_kellogg.* TO 'dms_readonly'@'%';
FLUSH PRIVILEGES;
```
- Schema: `dms_kellogg`
- User `dms_readonly` has `SELECT` only — use this for the Database tab connection and `/api/execute`
- Never connect with a write-privileged user from the AI backend

### `rag_chunk` table DDL
```sql
CREATE TABLE IF NOT EXISTS rag_chunk (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    file_name     VARCHAR(255) NOT NULL,
    file_type     VARCHAR(50),
    chunk_index   INT DEFAULT 0,
    content       TEXT NOT NULL,
    uploaded_at   DATETIME,
    INDEX         idx_rag_filename (file_name),
    INDEX         idx_rag_filetype (file_type),
    FULLTEXT INDEX idx_rag_fulltext (content)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
- `ENGINE=InnoDB`, charset `utf8mb4` — required for FULLTEXT + multi-byte characters
- Three indexes: filename lookup, filetype filter, and FULLTEXT search on `content`
- Run this once on the AI backend's MySQL instance before starting the service

## L3 Support — Report Issue Debugging Workflow

As L3 support, when a report screen has an issue:
- You run the query manually with the same parameters the user provided
- You trace the issue through each layer until you find the root cause
- You then fix it or escalate with full context

---

### Investigation Order (5 layers — always top to bottom)

```
LAYER 1  →  <ReportName>.xhtml             (UI — find the failing action/field)
LAYER 2  →  <ReportName>Bean.java          (Bean — collect params, call service)
LAYER 3  →  <ReportName>Service.java       (Service — build query params, business logic)
LAYER 4  →  query.xml                      (Query — SQL + parameter bindings)
LAYER 5  →  MySQL (manual)                 (Run query with real params, verify result)
```

---

### Layer-by-layer checklist

**Layer 1 — `currentstockweightreport.xhtml`**
- Find the button / filter / component that triggered the issue
- Identify the `action`, `actionListener`, or `value` EL binding on that component
- Note the exact method name it calls on the bean (e.g., `#{currentStockWeightReportBean.generateReport}`)

**Layer 2 — `CurrentStockWeightReportBean.java`**
- Go to the method identified in Layer 1
- Check: how are the screen input parameters (date, distributor, godown, etc.) collected?
- Check: are any parameters null, empty, or incorrectly defaulted?
- Note exactly what is passed into the service call

**Layer 3 — `CurrentStockWeightReportService.java`**
- Trace the service method called by the bean
- Check: how are bean parameters mapped to query parameters (e.g., into a `Map` or DTO)?
- Check: is there any filtering, date formatting, or conditional logic before the query runs?
- Note the query ID / method name that hits `query.xml`

**Layer 4 — `query.xml`**
- Find the query block by the ID called in Layer 3
- Read every `<if test="...">` condition — check if any condition silently excludes data
- Verify parameter names in `#{paramName}` match exactly what the service passes
- Check JOIN conditions, `WHERE` clause logic, and `GROUP BY` / `ORDER BY`

**Layer 5 — Manual DB verification**
- Extract the final SQL from `query.xml` (with all `<if>` conditions resolved for your test case)
- Run it in MySQL with the exact parameter values the user gave
- Compare the DB result to what the screen shows
- If DB result is correct → issue is in rendering/bean (Layers 1–2)
- If DB result is wrong → issue is in query/service (Layers 3–4)

### Starter diagnostic SQL (Current Stock Weight Report)

Replace only `<distr_br_code>` with the value from the reported issue.

```sql
-- 1. Verify distributor branch has pricing (SupplyChainMaster)
SELECT * FROM SupplyChainMaster
WHERE memberCode = '<distr_br_code>'
  AND cmpCode = 'Kellogg';

-- 2. Verify LOB is configured for Kellogg
SELECT * FROM lob_master
WHERE cmpCode = 'Kellogg';

-- 3. Verify stock exists for this branch
SELECT distrBrCode,
       COUNT(*)          AS row_count,
       SUM(saleableQty)  AS total_saleable,
       SUM(unSaleableQty) AS total_unsaleable
FROM StockOnHand
WHERE distrBrCode = '<distr_br_code>'
  AND cmpCode = 'Kellogg'
GROUP BY distrBrCode;

-- 4. Verify product hierarchy is mapped
SELECT p.prodCode, p.prodName, p.prodHierValCode, p.isActive
FROM Product p
WHERE p.cmpCode = 'Kellogg'
  AND p.isActive = 'Y'
LIMIT 20;
```
